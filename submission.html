Team Information

A file with the requested headers and an example row can be found here: teamName_members.csv

Each team should provide a CSV file with the name <<Team Name>>_members.csv with the following columns for each group member.

  • Name
  • E-Mail
  • ORCID iD
  • Affiliation
  • Affiliation2

Additional affiliation columns can be added as needed. The provided names and affiliations should reflect how participants wish the information to appear in a published report. Needed acknowledgements will be requested at a later stage.

Method Basics

A file with the requested headers and an example row can be found here: teamName_methods.csv

Each group should provide the following basic information in a CSV file with the name <<Team Name>>_methods.csv. Groups employing multiple methods should submit just one file with a different row for each method. Any method or method variation yielding distinct results (i.e., worth being analyzed separately) should be given a separate row.

The requested basic information is as follows:

  • Name: a full, descriptive name for each method, i.e. what would be used to refer to this method in a paper
  • Nick Name: a shorter, file-name friendly version (e.g. no spaces, special characters, etc.) to be incorporated into various submitted files
  • Contact: the e-mail of who to best contact for questions about this method; likely the person who will be doing the majority of the work implementing the method
  • Input: list all data products required by the method
  • Recalculated: list any data products that were re-calculated for the method—i.e., any data product for which the provided values were not used
  • Run On: the class of device was the method run on—e.g. Laptop, desktop, cluster (and please specify number of nodes, GPUs, etc.), other
  • Run Time: the order of magnitude unit it takes the method to finish running when using the device specified in the “Run On” column—e.g. seconds, hours, etc.

An example of how this data will be compiled can be found in Table 6 of the ESSP II report.

Method Details

We also request more detailed information on each method to be submited with the final results. These detailed questions focus on 1) a general description of the method, 2) the data requirements for the method, 3) the process of applying the method, and 4) the results of the method.

The questions to be answered are collected in the following file formats:

Results

RVs of Results

A file with the requested headers and an example row can be found here: dataSet_teamName_methodName_results.csv

All RVs should be given in units of m/s.

We ask that each method at minimum provide cleaned RVs, RV_C, and corresponding errors, eRV_C. These “cleaned RVs” are defined as publication ready RVs For instance, if you were conducting a planet search, the RV_C values are what that search would be performed on.

If available, methods should also provide activity RVs, RV_A, and corresponding errors, eRV_A. These “activity RVs” are defined as shifts attributed to stellar activity/variability or noise. For methods that do not return separate activity RVs, we will assume the activity RVs are the residuals from the cleaned RVs.

In other words, if we denote the provided RVs as RV_\odot, we make the assumption that:
RV_\odot = RV_C + RV_A
Where RV_C deviates from zero only due to injected planet shifts and intrinsic scatter.

For methods that derive any category of activity indicator, we also ask that the value of the indicators be provided along with errors for each value. Please be sure to specify the unit of the indicator in the column header if possible.

Each file should contain a Time [MJD] column that exactly mirrors the time stamps of the provided data set. We ask that results are returned for every observation provided, even if there exists outliers. Missing values should be left empty or filled in with the string NaN. There will be a place to discuss reasons why data may be missing or returning outliers in the detailed questions document for each method.

Planet Recovery

All provided cleaned RVs will be processed using the same planet-fitting pipeline in order to establish a consistent baseline comparison of planet parameters in the returned data.

Methods that employ joint or otherwise custom planet fitting are asked to also return their best-fit system parameters. If applicable, please also submit the posteriors for this planet fit. For these submissions, please submit a CSV file with the following for each planet:

  • K: semi-amplitude in m/s
  • P: period in days
  • t0: time of periastron in MJD

Planet Fit File Name: <<Data Set>>_<<Group Name>>_<<Method Name>>_planetFit.csv
Posterior File Name: <<Data Set>>_<<Group Name>>_<<Method Name>>_planetFitPosteriors.csv

Auxiliary Data

If the method is GP based, we ask that you submit your best-fit hyperparameters and any associated priors. Please describe what is contained in these files within the detailed method questions.
